# Bem-vindo(a) ao Antecipation Calculator

### Uma calculadora de antecipação para que as pessoas consigam saber quais valores receberão caso optem por antecipar o recebimento.

Esta é uma aplicação frontend feita em **React** utilizando a linguagem
**Typescript**. Como gerenciador de pacotes eu utilizei o **NPM**.

Alem das tecnologias já citadas acima, foi utilizada também as libs:

- React Router Dom
- Styled Components
- Axios
- React Hook Form
- Yup
- React Icons
- Story Book

Menos introdução e mais prática!

A seguir você encontrará como iniciar o projeto em ambiente de
desenvolvimento.

<br />

## Executando a aplicação em contexto local

<br />

- Primeiro faça o clone parao seu repositório local com o comando:

```shell
$ git clone git@github.com:thdev-matheus/frontend-antecipation-calculator.git
```

ou utilizando a CLI do github:

```shell
$ gh repo clone thdev-matheus/frontend-antecipation-calculator
```

- Instale os pacotes necessários para o funcionamento da aplicação com:

```shell
$ npm i
```

- Para inicializar a aplicação execute:

```shell
$ npm run start
```

## Contribuindo com o deseenvolvimento da aplicação

- Se for contribuir com a aplicação, pode criar uma nova branch a partir da develop para criar seus blocos de código:

```shell
& git checkout -b <nome_da_branch>
```

- Sempre mantenha sua branch atualizada então antes de codar, garanta que vai estar tudo 100% pra não haver conflitos (faça isso sempre):

```shell
$ git pull origin develop
```

- Commits nunca não demais então para não perder o seu progresso esteja sempre commitando e fazendo pushes para a sua branch

```shell
$ git add .
$ git commit -m "mensagem do commit"
$ git push origin <nome_da_branch>
```

- Por falar em commits, por aqui eu sigo o padrão do conventional commits usando as palavras chave: "feat: mensagem", "fix: mensagem", "refactor: mensagem". Então vamos manter dessa forma porque assi é mais organizado.

- Seu trabalho está pronto? É hora de mandar um pull request. Faça um pull da develop para atualizar o código e depois faça um push para a sua branch. No github, solicite com um pull request a inserção do seu código na aplicação e não esqueça de comentar as mudanças e o motivo delas. Os comentários podem ser em português ou em inglês, você decide.

- Leia o código antes de codar e se atente ao padrão de código e de arquitetura de arquivos. Queremos com certeza evitar uma salada de hard codes por aqui.

## Arquitetura de arquivos

<br />

<img src="https://i.ibb.co/9nFkRHH/Captura-de-tela-de-2022-11-16-03-39-28.png" />

Logo acima tem um exemplo de como a aplicação está arquitetada a nível de
componente numa separação de arquivos por função. Os arquivos são e serão
sempre:

- index.tsx - Responsável por guardar o componente jsx e sua lógica (se a lógica ficar demasiado grande pode-se cogitar escrever uma função utilitária na pasta utils e importá-la no componente).

- styles.ts - Como está sendo utilizado o styled-components na aplicação, o arquivo é responsável por guardar as tags HTML estilizadas para um determinado componente jsx.

- types.ts - Já que a aplicação está construída com a linguagem Typescript é imprescindível que se tenha cuidado de utilizar-se desse arquivo para guardar as tipagens do componente em si. Tanto por organização e limpeza, evitando poluir o código no arquivo do componente ou das tags estilizadas quanto por manutenção pois é bem fácil encontrar as tipagens do componente quando estão organizadas em um só lugar.

- <nome_no_componente>.stories.mdx - arquivo responsável escrever e integrar as funcionalidades e detalhes do componente à documentação.

## Padrão de código

Algumas regras precisam ser respeitadas para se ter um código homogêneo, bem
organizado, limpo e estruturado. são elas:

- Nome das funções, variáveis, Componentes e páginas em inglês e escritas no padrão camelCase (para funções e variáveis) e PascalCase (para componentes e páginas).
- Sintaxe de variável para as funções (sepre que possível tipar os retornos das funções) e componentes e exportação diretamente da variável (sem exportação default para evitar importações com nome errado). Por exemplo:

```typescript
export const NomeDoComponente = ({}: TipagemDasProps) => {
  return <h1>Hello, Kenzie!</h1>
}

export const nomeDaFunção = (par: Tipagem || {chave1, chave2}: Tipagem): TipagemRetorno => {
  // lógica
}
```

- Tipagens sempre devem ser interfaces e começar com a letra I e escritas em PascalCase.
